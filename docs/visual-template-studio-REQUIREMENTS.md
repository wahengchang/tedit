# 視覺模板編輯器（暫名 Visual Template Studio）— 需求文檔

> 給實作團隊的產品需求書。通用工具、不綁任何特定品牌。
> 暫名待定。本文件自給自足。

---

## 1. 產品一句話

一個**網頁版自由視覺設計編輯器**（Canva / Figma 那一類）：使用者自由設計版面，把畫面中**任意元件綁成具名變數**存成「模板」；之後用 **CLI 餵一份資料、headless 產出圖片**，同一模板換不同資料可重複產生。

## 2. 目標 / 非目標

**目標**
- 在瀏覽器自由設計圖片版面（拖拉、定位、分層、樣式）。
- 把設計存成可重複使用的**模板**，其中部分內容綁成**具名變數**。
- 用 CLI 指令：模板 ＋ 一份資料 → 一張圖片，可重複呼叫產生不同內容。
- **編輯器所見 與 CLI headless 產出，像素一致。**

**非目標**
- 不綁任何特定品牌 / 主題（使用者自己建立要的模板就夠）。
- v1 不做：多人協作、雲端帳號、動畫/影片、AI 生圖、一次吃整批資料表量產。

## 3. 核心概念

```
   設計（自由編輯）
        │  把任意元件綁成具名變數  {{title}} {{photo}} …
        ▼
     模板（設計 + 變數定義，存檔）
        │
        │  CLI： 模板 + 一份資料(填變數值)
        ▼
     一張 PNG     ← 重複呼叫、換資料 → 換內容、同版面
```

- **模板**：一份完整的自由設計（場景），其中某些元件被標記為變數插槽。
- **資料**：一組變數值（單筆）。
- **產生**：CLI 把資料灌進模板 → headless 渲染 → 輸出一張圖。重複是外部多次呼叫（v1 不內建批次表）。

## 4. 核心原則（最硬的約束）

**編輯器預覽 與 CLI headless 產出必須同像素。**
- 設計以一份**機器可讀的場景描述**（如 JSON）為單一真相。
- 編輯器與 headless 用**同一套渲染邏輯**畫同一份場景 → 任何機器、任何路徑，同結果。
- 反模式（排除）：編輯器一套畫法、headless 另一套 → 同模板出不同像素。

## 5. 使用流程

**A. 設計者（編輯器）**
1. 開編輯器，設定畫布尺寸與背景。
2. 自由加入文字 / 圖片 / 形狀，拖拉定位、調樣式、分層。
3. 把要隨資料變動的元件**綁上具名變數**（例：標題文字綁 `{{title}}`、某張圖綁 `{{photo}}`）。
4. 存成模板檔。

**B. 產生者（CLI / 自動化）**
1. 準備一份資料檔，填入各變數值。
2. `render <模板> <資料> -o out.png` → 一張圖。
3. 換資料、重複呼叫 → 同版面、不同內容。

## 6. CLI 設計

```
studio serve [--port <n>]
    # 起本地伺服器 + 開編輯器網頁；設計、綁變數、存模板

studio render <template> <data> [-o out.png]
    # headless 載入模板 + 灌入資料 → 渲染 → 輸出 PNG
    # 重複產生 = 外部以不同 data 多次呼叫

studio vars <template>        # （建議）列出模板有哪些具名變數，方便備資料
```

## 7. 資料模型

- **模板檔**：場景描述（所有元件、位置、層級、樣式、畫布尺寸/背景）＋ **變數定義**（哪個元件的哪個屬性綁了哪個變數名）。由編輯器產生，格式建議 JSON（場景圖機器產出、不手編）。
- **資料檔**：變數名 → 值 的對應。建議支援好手編的格式（YAML/JSON）。
  ```yaml
  title: "這次的標題"
  photo: ./images/a.png      # 圖片變數可給路徑或 URL
  ```
- 變數可綁的屬性至少包含：文字內容、圖片來源；其餘屬性（顏色、可見性…）可由實作團隊評估擴充。

## 8. v1 編輯器元件範圍

| 元件 | v1 能力 |
|------|---------|
| 文字 | 字體、大小、顏色、對齊、行高；內容可綁變數 |
| 圖片 | 貼上 / 拖放 / 路徑載入；縮放裁切；來源可綁變數 |
| 形狀 | 矩形、圓、線/分隔線；填色、描邊 |
| 背景 | 畫布尺寸、背景色 / 背景圖 |
| 通用 | 拖拉定位、分層（z-order）、選取、刪除、複製 |

> 更多 Canva / Figma 式功能（群組、對齊輔助線、字型管理、效果、復原/重做…）由實作團隊探索現有工具後，於 v1 後逐步擴充。

## 9. 變數綁定規格

- **任意元件**的可變屬性都能綁一個**具名變數**（例：文字元件的「內容」綁 `{{title}}`）。
- 同一變數名可被多個元件共用（一處資料、多處反映）。
- 未綁變數的元件＝固定內容，每次產出都一樣。
- 資料未提供某變數時，行為（用設計時的預設值 / 留空 / 報錯）由實作團隊定義並寫進文件。

## 10. 開放技術決策（交實作團隊探索定案）

| 項 | 約束 / 方向 | 備註 |
|----|-------------|------|
| 渲染基底 | **必須滿足 §4：編輯器預覽 == headless 產出，同像素** | 候選：Canvas 函式庫（fabric.js / Konva，場景存 JSON、編輯器與 headless 同 lib 重畫）、HTML/CSS 絕對定位 DOM、SVG 場景。團隊自行評估後選定並記錄理由。 |
| 編輯器功能廣度 | 先做 §8 核心，再參考 Canva/Figma 擴充 | 探索期可先盤點現成工具的互動範式 |
| 字體處理 | 需保證跨機/離線一致 | 建議 self-host / 打包字體 |
| 模板/資料檔格式細節 | §7 方向為準 | 場景建議 JSON、資料建議 YAML/JSON |

## 11. 技術棧（已定方向）

- **Node + TypeScript**（編輯器與 CLI 同語、共用渲染邏輯）。
- **headless 瀏覽器**（如 Playwright）負責 `render` 出圖。
- 本地 http server 供 `serve` 模式。

## 12. 規模與里程碑（粗估）

這是**數週級**的專案（全自由視覺編輯器本身就是主要工程）。建議里程碑：

1. **渲染核心** — 選定渲染基底；能載入一份場景 JSON、在編輯器與 headless 各畫一次、驗證同像素。
2. **render CLI** — `render <template> <data>` 出 PNG。
3. **變數綁定** — 場景內元件可綁具名變數；資料灌入生效；`vars` 列變數。
4. **編輯器 v1** — 文字 / 圖片 / 形狀 / 背景 + 拖拉、分層、選取、存模板。
5. **serve 整合** — 編輯器存出的模板能被 render 直接吃；打磨字體、預設值、尺寸。
6. **擴充期** — 參考 Canva/Figma 補功能。

## 13. v1 明確不做

- 一次吃整批資料表量產（v1 是單筆、外部重複呼叫）。
- 多人協作 / 雲端 / 登入。
- 動畫、影片、AI 生圖。
- 任何內建品牌主題。
